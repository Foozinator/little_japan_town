# CLAUDE.md — Izakaya shop

**Before making changes: read this file, then `coding_standards.md` at the workspace root.**

## What this is
Firmware for the izakaya shop — a laser-cut Japanese shop miniature containing one white ceiling LED, a NeoPixel accent strip, and a TTP223 capacitive touch sensor on the roof. This shop is a **child node** on the RS485 bus and speaks Modbus RTU slave to the cornerstone.

## Target
- **Board:** Arduino Nano ATmega328P (genuine)
- **PlatformIO env:** `nano`
- **Framework:** arduino
- **Serial (USB):** 115200 baud, debug only
- **Bus:** RS485 via SoftwareSerial at 9600 baud; pins in `pins.h`

## Identity
- **Shop name:** `ramen` — stored in EEPROM, read at boot
- **Modbus slave address:** `0x02`
- **EEPROM layout:**
  - Offset 0: shop name, 16 bytes, null-terminated ASCII
  - Offset 16: Modbus address, 1 byte
  - Remainder unused

If either EEPROM field reads as `0xFF` (unprogrammed), the firmware halts with a serial error and does not attempt to join the bus.

## Peripherals, in registration order
Construction order in `main.cpp` determines the Modbus register layout. **Do not reorder.**

1. `Touch touch`           — TTP223 capacitive input
2. `WhiteLight whiteLight` — PWM-driven white ceiling LED
3. `Neopixel neopixel`     — 8-pixel WS2812B accent strip
4. `Bus bus`               — Modbus slave layer (always last)

## Current register map
Holding registers, generated by `Bus::begin()` from the registration list above:

| Offset | Peripheral | Field         | Type   | Access |
|--------|------------|---------------|--------|--------|
| 0x00   | touch      | pressed       | uint8  | read   |
| 0x01   | whiteLight | brightness    | uint8  | r/w    |
| 0x02   | neopixel   | red           | uint8  | r/w    |
| 0x03   | neopixel   | green         | uint8  | r/w    |
| 0x04   | neopixel   | blue          | uint8  | r/w    |

The descriptor region (read by the cornerstone at discovery) lives at input registers `0xF0+` and is built automatically from the same registration list.

## Hardware quirks on this board
- TTP223 is in **momentary** mode — `CI` is HIGH only while touched. Configured by leaving pads TP1/TP2 unbridged on the sensor board.
- White LED is driven through a 2N7000 MOSFET, active-HIGH. Resistive load, no flyback needed.
- NeoPixel strip is data-in only, no return path. Corrupted frames are not recoverable until the next refresh.
- Powered from the 5V RS485 bus (JST-XH). No local regulator on this board.
- USB port is accessible through the shop's rear access flap without disassembly.

## Build and flash
```
pio run                      # build
pio run -t upload            # flash over USB
pio device monitor           # serial debug
```

## Hard rules (short form)
Full version in `coding_standards.md`. The ones that come up first:

1. Do not modify `pins.h`.
2. Do not reorder peripheral construction — it rewrites the register map.
3. No `delay()` in `loop()`.
4. No dynamic allocation after `setup()`.
5. `String` class is forbidden.
6. If a change requires breaking any of these, stop and surface the conflict.

## Known gaps / open questions
- Touch handling: currently exposes raw "pressed" state. Whether the shop should implement local toggle/debounce or leave that to the cornerstone is not yet decided.
- NeoPixel: plain RGB triplet. No animations, no brightness scaling. Likely to change.
- Bus join sequence and timeout behavior on missing cornerstone is not yet specified.
