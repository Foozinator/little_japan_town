# CLAUDE.md — Choshi no Shitamachi

**Before making changes: read this file, then `coding_standards.md` at the workspace root.**

## What this is
Firmware for **Choshi no Shitamachi** — a laser-cut Japanese izakaya miniature. This shop is a **child node** on the RS485 bus and speaks Modbus RTU slave to the cornerstone.

## Status: under active construction
The physical shop and its circuit are being assembled in parallel. Expect:
- `pins.h` values are **provisional** and will change as wiring settles. The "frozen" rule in the standards doc does not yet apply to this project — a human is still editing pin assignments during construction.
- Peripheral count and types may change. The current list below is the working plan, not the final configuration.
- Firmware is scaffold-only; not yet on the bus.

When the physical build and wiring are finalized, this section gets replaced with a `## Status: deployed` block and `pins.h` becomes frozen per the standards doc.

## Target
- **Board:** Arduino Nano ATmega328P (genuine)
- **PlatformIO env:** `nano`
- **Framework:** arduino
- **Serial (USB):** 115200 baud, debug only
- **Bus:** RS485 via SoftwareSerial at 9600 baud; pins in `pins.h`

## Identity
- **Shop name:** `choshi` — stored in EEPROM, read at boot
- **Modbus slave address:** `0x02` *(provisional — confirm against cornerstone's assignment table before first power-on)*
- **EEPROM layout:**
  - Offset 0: shop name, 16 bytes, null-terminated ASCII
  - Offset 16: Modbus address, 1 byte
  - Remainder unused

If either EEPROM field reads as `0xFF` (unprogrammed), the firmware halts with a serial error and does not attempt to join the bus.

## Peripherals, in registration order
Construction order in `main.cpp` determines the Modbus register layout. **Do not reorder.**

1. `Touch touch`            — TTP223 capacitive input on the roof
2. `LampWhite lampWhite`    — PWM-driven white LED *(interior lighting, planned)*
3. `LampAccent lampAccent`  — NeoPixel strip *(chōchin / accent lighting, planned)*
4. `Bus bus`                — Modbus slave layer (always last)

Lamp assignments are provisional. If the shop ends up with a different lamp configuration (more lamps, different types, or one lamp only), update this list and the register map below together.

## Current register map (planned)

Holding registers, generated by `Bus::begin()` from the registration list above. Follows the standard touch and lamp conventions defined in `coding_standards.md`.

| Offset       | Peripheral  | Field                               | Type     | Access |
|--------------|-------------|-------------------------------------|----------|--------|
| 0x00         | touch       | rising-edge counter                 | uint8    | read   |
| 0x01 – 0x06  | lampWhite   | mode, target_r/g/b, fade_ms_lo/hi   | 6× uint8 | r/w    |
| 0x07 – 0x0C  | lampAccent  | mode, target_r/g/b, fade_ms_lo/hi   | 6× uint8 | r/w    |

Descriptor region (read by cornerstone at discovery) lives at input registers `0xF0+` and is built automatically from the same registration list.

## Hardware quirks on this board
*Provisional — to be confirmed during construction.*

- **Touch:** TTP223 in momentary mode (CI HIGH while touched). Configured by leaving pads TP1/TP2 unbridged on the sensor board.
- **White LED brightness-from-RGB fallback:** `lampWhite` is a single-channel LED. When it receives a non-white `target` color, it lights at `max(r, g, b)` as plain PWM brightness. Theme IDs that call for white or ongoing animation (COOL_WHITE, WARM_WHITE, CANDLE, FLUORESCENT) behave as specified in the standards doc; other themes fall back to STATIC brightness.
- **NeoPixel accent (`lampAccent`):** Pixel count TBD during construction — will go into `pins.h` as `LAMP_ACCENT_COUNT`. Data-in only, no return path; corrupted frames are not recoverable until the next refresh.
- **Power:** 5V from the RS485 bus (JST-XH). No local regulator.
- **Programming access:** USB port location during construction is still being determined. OTA is not available on the Nano, so physical USB access is required for all flashing.

## Hard rules (short form)
Full version in `coding_standards.md`. The ones that come up first:

1. Do not modify `pins.h` **once construction is complete** (see Status above). During construction, a human may edit it; agents still should not.
2. Do not reorder peripheral construction — it rewrites the register map.
3. No `delay()` in `loop()`.
4. No dynamic allocation after `setup()`.
5. `String` class is forbidden.
6. Lamp blocks are written atomically with Modbus function 0x10; don't split them.
7. If a change requires breaking any of these, stop and surface the conflict.
